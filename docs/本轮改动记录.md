# 本轮改动记录

日期：2026-05-16

## 已调整

### 1. 工具流程文案

- 将 `save_items` 的工具结果文案从“待确认”改为“已整理成记录草稿”。
- 将识别建议卡片的主按钮从“确认”改为“保存”。

涉及文件：

- `apps/mobile/src/components/AgentWorkflow.js`
- `apps/mobile/src/components/SuggestionCard.js`

原因：

- `save_items` 的真实语义是提交一份可编辑、待用户保存的记录草稿，不是模型再次等待用户确认。
- “保存”比“确认”更符合用户实际动作。

### 2. 空空间位置查询误报

- 修复 `list_positions` 在空间存在但没有位置时返回“没有找到空间”的问题。
- 现在逻辑区分：
  - 空间存在但无位置：返回空数组，前端显示“暂无位置”。
  - 空间不存在：返回“没有找到空间”。

涉及文件：

- `apps/api/src/store.js`
- `apps/api/src/tools.js`

原因：

- `list_spaces` 能返回空间，但 `list_positions` 查不到位置时，旧逻辑直接把空结果当成空间不存在。

### 3. 识别提示词中的数据结构边界

- 在模型提示词中补充 `space / position / item` 的定义：
  - `space`：房间或家里的大区域，如客厅、卧室、厨房、玄关、走廊。
  - `position`：空间里的真实家具、收纳点、台面、地面或局部区域，如电视柜、茶几、书桌、走廊地面、客厅一角。
  - `item`：用户日后会问“XX在哪”的具体物品。
- 明确禁止模型把照片状态、任务状态、占位词当成位置名。
- 在当前家庭结构摘要里，如果已有位置名属于历史占位名，会标注为“历史占位名，勿复用”。

涉及文件：

- `apps/api/src/agent.js`

原因：

- 数据库里已有 `客厅 / 待识别图片` 这类历史占位位置，模型会因为“优先复用已有位置”而继续沿用。
- 本次按要求只做模型层约束，没有在 `save_items` 工具层新增占位名校验。

## 当前问题记录

### `update_item` 工具语义不够清晰

当前模型只拿到一个更新类工具：

- `update_item`：改物品名、改物品描述、把物品移动到某个空间/位置。

它不能做：

- 重命名空间。
- 重命名位置。

因此当用户说“把 ceshi1 改成空位、待识别图片改成证件区”时，这是位置重命名需求，但模型容易误用 `update_item`，把 `ceshi1` 当成物品名传给工具。

建议后续拆分工具：

- `update_item`：只改物品名和描述。
- `move_item`：移动物品当前位置，最好明确是移动最新记录还是全部历史记录。
- `rename_position`：重命名位置。
- `rename_space`：重命名空间。

### 运行中后端可能不是最新代码

浏览器里出现的错误：

```text
invalid input syntax for type uuid: "ceshi1"
```

说明当前运行中的 API 进程可能仍是旧逻辑，把普通名称先按 UUID 查询。

本地代码里已有按 UUID 格式判断后再查 ID 的保护逻辑；如果运行服务没有自动重载，需要重启 API 服务后才会生效。
