# 多语言支持开发计划

> **状态（2026-07-12）：已落地。** 后端（1.1 AI 语言 + 1.2 错误码）已 push 部署；前端（strings.js zh/en + 全部文案替换 + 错误码映射 + spaceCovers 英文词补充）已完成，等 EAS 发版生效。剩余：模拟器双语言回归 + EAS build + TestFlight 验证（见"三、验证"前端部分）。
> 实现备注：纯照片场景仅靠 system prompt 注入语言不够（analyze 的中文包装文字会被模型当成用户语言），已在无文字请求里显式注入"用设备语言（xx）回复"，实测生效。

背景:App 已多区上架,线上已出现非中文系统语言的用户,但 App 内 UI、AI 回复、错误提示全是中文。语言判定统一跟随**设备系统语言**(不跟 App Store 分区):前端用 `expo-localization`,后端用请求头 `Accept-Language`(iOS 网络层自动携带,server.js 已在读它做统计,前端无需任何改动)。

本计划为一次完整开发:后端(AI 语言 + 错误码)+ 前端(UI 英文化),后端部分向后兼容可先行 push,前端随一次 EAS 发版收尾。ASC 商店页英文化是后台操作不在此列;locationRules 英文词表看英文用户量再定。

## 一、后端

### 1.1 AI 回复跟随用户语言

1. **server.js — 解析请求语言**
   - 抽 `parseLanguage(req)`:取 `Accept-Language` 首个 locale 的主语言标签(`en-US,en;q=0.9` → `en`),取不到返回 `zh`;只给模型干净的语言码,不传原文
   - `/agent/analyze`(约 928 行)和 `/agent/query`(约 1097 行)调用 `runAgent` 时传入 `userLanguage`

2. **agent.js — prompt 注入语言指令**
   - `runAgent`(约 408 行)签名加 `userLanguage` 参数
   - system prompt 拼接处(约 419 行)追加一句:**优先使用用户消息的语言;用户没有文字输入时,默认用设备语言({userLanguage})**。命名会自然跟随对话语言,不单独指示
   - 搜索规则追加(与语言指令无关,解决跨语言检索):`search_items` 未命中时,把关键词翻译成"当前家庭数据"中已有名称所用的语言再搜一次,仍未命中才走既有的"没找到"排查流程

注意:
- 用每次请求的头判定,不要用 users 表的 `last_language`(统计快照,10 分钟节流,切语言会滞后)
- 纯拍照识别(无文字输入)是核心受益场景:照片里没有语言信号,没有这段注入模型会跟着中文 prompt 回中文
- 对话是 previousResponseId 链式的,历史中文上下文可能让模型偶尔"惯性"回中文,prompt 写明确即可,不为此重构

### 1.2 错误码

- server.js 所有 `sendJson(res, 4xx, { error: '中文' })` 增加 `code` 字段(如 `INSUFFICIENT_CREDITS`、`DAILY_LIMIT_REACHED`、`EMPTY_NAME`、`QUERY_TOO_LONG`,约 70 处、归并后预计 20 个左右错误码),**保留原 `error` 中文文案**——老版本 App 继续显示 `error`,新版本优先按 `code` 查 strings.js,查不到 fallback 到 `error`

后端两项都向后兼容,可以在前端完成前先 push 上线(1.1 立即对海外用户生效)。

## 二、前端

### 2.1 文案基础设施

- 新建 `apps/mobile/src/strings.js`:
  - `zh` / `en` 两个扁平字典 + `t(key, params)` 函数(支持 `{count}` 这类简单插值)
  - 用 `expo-localization` 的 `getLocales()[0].languageCode` 选语言,非 `zh` 一律走 `en`

### 2.3 付费墙专项

### 2.4 错误码映射

- 前端统一的错误处理处优先按后端返回的 `code` 查 strings.js,查不到 fallback 显示 `error` 字段

### 2.5 已知难点

- **spaceCovers.js**:中文空间名映射封面图,英文空间名("Living Room")匹配不上会落到默认封面——补一份英文关键词映射

- **布局回归**:英文文案普遍比中文长,按钮/标签可能溢出换行,付费墙和 SuggestionCard 重点检查

## 三、验证

后端(本地起 API 后):
1. `Accept-Language: en` 调 `/agent/query`,确认英文回复
2. `Accept-Language: en` 调 `/agent/analyze` 传测试图,确认英文回复、suggestion 里空间/位置/物品名是英文
3. `Accept-Language: zh-CN` 回归中文链路无变化
4. 英文关键词搜中文库存物品(如 scissors → 剪刀),确认翻译重试生效
5. 4xx 响应确认同时带 `code` 和中文 `error`

前端:
1. Web 开发模式(`npx expo start --web`)浏览器改语言过一遍全部页面
2. 模拟器分别设中文/英文系统语言,过登录 → 引导 → 拍照识别 → 确认保存 → 查询 → 付费墙全流程
3. 中文版完整回归(290 处替换,防止改坏现有文案)
4. 触发余额不足(403)、每日限额(429)、空名称(400),确认错误码映射和 fallback 都正常
5. `npx eas-cli build --profile preview --platform ios` → TestFlight 真机双语言验证
