# FindIt 两套方案对比

## 方案概览

| 维度 | PRD.md（方案 A） | light-vision-mvp（方案 B） |
|------|-----------------|--------------------------|
| 定位 | 完整产品规划，含 P0/P1/P2 | 最小闭环 MVP，只验证核心价值 |
| 核心理念 | AI 自动识别房间结构 + 物品 | 用户手动建位置，AI 只识别物品 |

---

## 1. 功能范围

| 功能 | 方案 A | 方案 B |
|------|--------|--------|
| 照片上传 | P0 | 做 |
| 视频上传 | P1（抽帧分析） | 明确不做 |
| AI 识别物品 | P0 | 做 |
| AI 识别房间类型 | P0（自动推断房间） | 明确不做（容易错） |
| 用户确认 AI 结果 | 有，但未强调流程 | 核心流程，pending → confirmed |
| 自然语言问答 | P1 | 第 3 阶段做 |
| 物品分类标签 | P1 | 做（category + aliases） |
| 家庭共享 | P2 | 明确不做 |
| 语音输入 | P2 | 不做 |
| 物品边框标注 | 未提及 | 明确不做 |
| 搜索历史 | P1 | 不做 |

**差异点：** 方案 B 砍掉了视频、房屋结构分析、家庭共享等功能，只保留拍照 → 识别 → 确认 → 查询的最小闭环。方案 A 功能更全但 MVP 阶段工作量更大。

---

## 2. 数据模型

### 方案 A：4 层层级结构
```
users → houses → rooms → locations → items ← item_photos → photos
```
- 6 张核心表 + 1 张关联表
- 物品直接绑定到 location，没有"观察记录"概念
- 物品位置变化时需要更新 location_id

### 方案 B：扁平结构 + 观察记录
```
users → locations → items ← item_observations → media_assets
```
- 4 张核心表
- 引入 `item_observations` 表，记录每次"在某位置看到某物品"
- 物品可被多次观察到，查询时取最近一次 confirmed 记录
- `items` 表有 `aliases` 字段，天然支持同义词搜索

### 对比

| 维度 | 方案 A | 方案 B |
|------|--------|--------|
| 层级深度 | houses → rooms → locations（3 层） | locations（1 层，用户自定义名称） |
| 位置建模 | AI 推断房间类型 + 用户创建子位置 | 用户直接创建，如"卧室床头柜左抽屉" |
| 物品位置关系 | 1 对 1（item.location_id） | 多次观察（item_observations） |
| 位置历史 | 无，更新即覆盖 | 有，每次观察都是一条记录 |
| 搜索支持 | `tsvector` 全文索引 | `aliases` 数组 + GIN 索引 |

**结论：** 方案 B 的数据模型更适合 MVP。`item_observations` 设计更贴合实际（物品会被移动），aliases 对搜索召回很关键。方案 A 的 houses → rooms → locations 层级对个人用户偏重。

---

## 3. AI 架构

### 方案 A：模型直连
```
API 接口 → 直接调 Azure OpenAI → 返回结果
```
- Prompt 散落在各 API 接口中
- 没有统一的模型调用封装
- 自然语言查询方案：全量物品数据塞进 GPT 上下文

### 方案 B：AI Agent Service 层
```
API 接口 → AI Agent Service → Azure OpenAI → 结构化输出
```
- 3 个明确能力：`analyze_photo` / `parse_find_query` / `compose_answer`
- 统一封装 Prompt、JSON 校验、错误兜底
- 自然语言查询方案：模型提取关键词 → SQL 查库 → 模型组织回答

### 自然语言查询实现对比

| 维度 | 方案 A（全量上下文） | 方案 B（提取关键词 → SQL） |
|------|---------------------|--------------------------|
| 实现复杂度 | 低，一次 GPT 调用 | 中，3 步流程 |
| 物品 < 500 时 | 完全够用 | 也够用，但多一次模型调用 |
| 物品 > 1000 时 | 上下文可能超限或成本高 | 不受影响，SQL 查询无上限 |
| 模糊匹配能力 | 强，GPT 天然理解同义词 | 依赖 aliases 预存的同义词 |
| 每次查询 token 成本 | 高（全量数据每次都发） | 低（只发关键词和候选记录） |
| 幻觉风险 | 低（数据在上下文中） | 更低（模型不接触位置数据，只组织回答） |
| 可追溯性 | 弱（GPT 黑盒回答） | 强（SQL 查询可审计） |

**结论：** MVP 阶段物品少，方案 A 的全量上下文更简单。但方案 B 的三步方案更稳健、成本更可控，且 Agent Service 封装对后续维护有利。

---

## 4. Prompt 设计

| 维度 | 方案 A | 方案 B |
|------|--------|--------|
| 图片识别 Prompt | 简略，让模型同时猜房间类型 + 位置 + 物品 | 聚焦物品识别，不猜房间，有 aliases 和 uncertain_items |
| 查询 Prompt | 无独立设计（全量塞上下文） | 拆成 parse + compose 两步，各有独立 Prompt |
| JSON 失败兜底 | 未考虑 | 有自动修复逻辑 + 兜底提示词 |
| Prompt 文档化 | 内嵌在 PRD 中 | 独立 `model-prompts.md` 文件 |

**结论：** 方案 B 的 Prompt 设计更成熟。aliases 支持、uncertain_items 分类、JSON 修复兜底都是实战经验。

---

## 5. 后端架构

| 维度 | 方案 A | 方案 B |
|------|--------|--------|
| 计算层 | Azure Functions (Serverless) | Azure Container Apps（推荐）或 Functions |
| 数据库 | Azure Cosmos DB (PostgreSQL) | Azure PostgreSQL Flexible Server |
| 密钥管理 | 未提及 | Azure Key Vault |
| API 数量 | 20+ 接口 | 12 接口 |
| 页面数量 | 8 页 | 5 页 |

**结论：** 方案 B 接口更少、页面更少，开发量更小。方案 B 推荐 Container Apps 也更务实（Functions 冷启动对 AI 调用体验不好）。

---

## 6. 开发策略

### 方案 A：4 个 Phase，功能并行推进
```
Phase 1: 骨架 + 认证 + 上传
Phase 2: AI 识别 + CRUD
Phase 3: 搜索 + UI
Phase 4: 测试上线
```

### 方案 B：3 个阶段，逐步引入 AI
```
阶段 1: 不用 AI，跑通数据闭环（手动添加物品）
阶段 2: 接入视觉模型（自动识别 + 用户确认）
阶段 3: 自然语言查询
```

**结论：** 方案 B 的分阶段策略更稳。先不用 AI 跑通数据流，确保基础功能可用；再接 AI 锦上添花。方案 A 第一步就依赖 AI，风险更高。

---

## 7. 综合推荐

| 维度 | 推荐 | 原因 |
|------|------|------|
| 功能范围 | 方案 B | 砍得更狠，MVP 聚焦 |
| 数据模型 | 方案 B | item_observations + aliases 更实用 |
| AI 架构 | 方案 B | Agent Service 封装 + 三步查询更稳健 |
| Prompt 设计 | 方案 B | 更完整，有兜底 |
| 开发策略 | 方案 B | 先跑通数据再接 AI，风险低 |
| 自然语言（MVP 初期） | 方案 A | 全量上下文更简单，物品少时够用 |

### 建议：以方案 B 为主体，吸收方案 A 的优点

1. **整体框架用方案 B** — 扁平位置模型、item_observations、AI Agent Service、5 页 MVP。
2. **自然语言查询先用方案 A 的全量上下文**  — MVP 初期物品少，一次调用更简单。物品超 500 后再切到方案 B 的三步方案。
3. **保留方案 A 的 P1/P2 路线图** — 视频、家庭共享、语音输入作为后续迭代方向。
4. **Prompt 和兜底逻辑用方案 B** — aliases、uncertain_items、JSON 修复都要保留。
